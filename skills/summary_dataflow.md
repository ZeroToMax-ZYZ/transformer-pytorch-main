# Skill: 整理数据流文档（summary_dataflow）

当用户要求"整理数据流"、"梳理 shape"、"画数据流图"或类似请求时，按以下规范输出。

---

## 核心原则

**用具体数字说话，不用抽象符号堆砌。**

目标读者是"正在读代码、想搞清楚每一步 shape 变化"的人，不是需要 API 参考手册的人。文档应该让人**跟着走一遍就能理解**，而不是需要对照代码逐行验证。

---

## 结构模板

### 1. 开头：超参数速查表

用一个表格列出所有符号及其典型值。后续文档中统一使用这些符号。

```markdown
| 符号 | 含义 | 典型值 |
|------|------|--------|
| B | batch_size | 4（动态） |
| S | 源序列长度 | 12（变长） |
| ... | ... | ... |
```

### 2. 按阶段编号，从数据源头到 loss

每个阶段对应数据流中的一个逻辑步骤，用 `## 阶段 N：名称（代码位置）` 格式。

阶段划分参考：
- 阶段 0：原始数据 → DataLoader
- 阶段 1：Batch 构造（mask、shift right）
- 阶段 2：Embedding
- 阶段 3：Encoder
- 阶段 4：Decoder
- 阶段 5：Generator / 投影头
- 阶段 6：Loss 计算

### 3. 每个阶段的写法

#### 3.1 先给代码片段（简短）

只贴关键的一两行，让读者知道"调的是什么函数"。

```python
src_mask = (src != pad_idx).unsqueeze(1)   # (B, S) → (B, 1, S)
```

#### 3.2 再给具体数字示例

**必须用具体数值画出来**，不要只写 `(B, S)`。

```
示例 (B=2, S=6):
src = [[156, 4821,  87, 3012,   2,   0],
       [ 72,  533, 118,    2,   0,   0]]
```

#### 3.3 用 ASCII 图解释复杂结构

mask、注意力矩阵、下三角矩阵等，用方框图或网格图表示：

```
causal_mask (下三角):
┌─────────────────┐
│ True  False False│
│ True  True  False│
│ True  True  True │
└─────────────────┘
```

#### 3.4 标注"作用"或"物理意义"

每一步操作后，用一句话解释**为什么这么做**，而不是只说**做了什么**。

```markdown
**物理意义**: 训练时用 Teacher Forcing。Decoder 看到 `<bos>` 时应该预测出 `89`。
```

#### 3.5 明确标注输出 shape 和 dtype

每个阶段结束时，用代码块或表格明确写出输出：

```
src_embed_out:  (B, S, 512)   dtype=float32
```

## 特殊图形规范

### 流程树（模块内部数据流）

用 `├──`、`│`、`└──` 画树状图，展示模块内部的逐层计算。每一步标注 shape 变化。

```
输入 x: (B, S, 512)
    │
    ├── SublayerConnection[0] ── Self-Attention
    │   │
    │   ├── LayerNorm(x)         → (B, S, 512)
    │   ├── MultiHeadedAttention
    │   │       ├── W_q(norm_x)  → (B, S, 512)
    │   │       ├── reshape → (B, S, 8, 64)
    │   │       ├── scores = Q @ K^T / √64  → (B, 8, S, S)
    │   │       └── ...
    │   │
    │   └── x = x + dropout(attn_out)   → (B, S, 512)
    │
    └── 输出 x: (B, S, 512)  → 送入下一层
```

### 全流程总览图

在文档末尾，用一个大的 ASCII 图把所有阶段串起来。用方框 + 箭头展示数据流向：

```
┌────────────────────────────┐
│  BPE 文本                   │
│  "A dog is running"        │
└──────────────┬─────────────┘
               ↓
┌──────────────┴─────────────┐
│  DataLoader                 │
│  src: (B, S)    int64       │
└──────────────┬─────────────┘
               ↓
           ...继续...
```

### 变换箭头

shape 变化用箭头符号：

```
src: (B, S)  ──Embedding──→  (B, S, 512)
```

在文档末尾加一个汇总表，列出所有关键 shape 变化：

## 写作禁忌

1. **不要只列符号不给数字**。`(B, S, d_model)` 是必要的，但必须同时给出"示例 B=2, S=6, d_model=512"的具体张量。

2. **不要用大段代码替代解释**。代码片段只用来指明"调的是哪个函数"，shape 变化用文字/图/表来表达。

3. **不要省略中间 shape**。比如多头注意力要写清楚 `(B, S, 512)` → reshape → `(B, S, 8, 64)` → transpose → `(B, 8, S, 64)`，不要一步跳过。

4. **不要忽略 mask 的构造过程**。mask 是 Transformer 中最容易出 bug 的地方，必须用 ASCII 图画清楚 causal_mask、pad_mask、以及它们如何组合。

5. **不要写成 API 文档风格**。避免"该函数接受以下参数..."这种句式。用"这一步把 `(B, S)` 的 token id 查表变成 `(B, S, 512)` 的稠密向量"这种面向数据流的描述。

6. **不要遗漏 dtype**。shape 之外，标注 `int64` / `float32` / `bool` 等类型信息，特别是 mask 的 `bool` 类型。

## 与 docs/ 目录下参考文档的区别

`docs/` 目录下的文档（如 `BPE_DATAFLOW_TENSOR_SHAPES.md`）适合做**API 级参考**，特点是：
- 穷举所有分支（token-budget vs 固定 batch、分布式 vs 单机）
- 列出所有中间变量的精确 shape
- 包含实测 shape 示例
- 适合作为 debug 手册

本 skill 产出的文档适合做**学习/理解**用，特点是：
- 只走一条主路径，不穷举分支
- 用具体数字和 ASCII 图帮助理解
- 强调"为什么"（物理意义）而非"是什么"（API 说明）
- 适合第一次读代码时跟着走一遍

两种文档互补，不需要合并。
